# 연구 과제 관련 논문(컨테이너의 수직 탄력성 효과)

> **논문**: Employing Vertical Elasticity for Efficient Big Data Processing in Container-Based Cloud Environments  
> **저자**: Jin-young Choi, Minkyoung Cho, Jik-Soo Kim (명지대학교, 2021)  

---

## 배경 개념 정리

### 하이퍼바이저(Hypervisor) 기반 가상화

물리 서버 위에 **가상 머신(VM)** 을 올려 여러 OS를 동시에 실행하는 기술
대표적으로 VMware vSphere, KVM 등이 있음

- VM마다 **Guest OS 전체를 포함**하기 때문에 무겁고 자원 소모가 큼
- 자원(CPU·메모리)이 **처음에 고정 할당**되어, 실행 중에 변경하려면 VM을 일시 정지해야 함
- 따라서 **수평 탄력성**(scale out/in, VM 개수를 늘리거나 줄이는 것)만 가능

### 컨테이너(Container) 기반 가상화

Guest OS 없이 **실행 파일과 종속성만** 패키징하여 실행하는 경량 가상화 기술.  
대표적으로 Docker, Linux Containers가 있음.

- Guest OS가 없어 **훨씬 가볍고 빠름**
- 실행 중단 없이 CPU·메모리를 **런타임에 동적으로 조정**가능(**수직탄력성**)

### 수직 탄력성) vs 수평 탄력성

| 구분 | 방식 | 설명 | 주로 사용되는 기술 |
|------|------|------|-------------------|
| **수평 탄력성** | Scale Out / In | 컨테이너(또는 VM)의 **개수**를 늘리거나 줄임 | 하이퍼바이저(VM), AWS Auto Scaling |
| **수직 탄력성** | Scale Up / Down | 하나의 컨테이너에 할당된 **자원의 양**을 늘리거나 줄임 | Docker 컨테이너 |

> 이 논문의 핵심은 **수직 탄력성**을 빅데이터 워크로드에 적용하면 자원 낭비를 줄이고 전체 처리량을 높일 수 있다는 것

### 주요 용어 정리

| 용어 | 설명 |
|------|------|
| **Makespan** | 전체 작업이 완료되기까지 걸린 총 시간 |
| **cAdvisor** | Google이 개발한 Docker 컨테이너 실시간 리소스 모니터링 도구 |
| **HiBench** | Intel이 관리하는 빅데이터 벤치마크 모음 (총 29개 워크로드) |
| **Terasort** | 대용량 데이터 정렬 벤치마크 (Micro 카테고리) |
| **K-means** | 클러스터링 머신러닝 알고리즘 벤치마크 (ML 카테고리) |
| **PageRank** | 웹 페이지 순위 계산 알고리즘 벤치마크 (WebSearch 카테고리) |

| 항목 | 설명 |

## Section 4.2 — 실험 결과

### 단일 컨테이너 실험 (Baseline vs Adaptive)

- Adaptive 방식에서 **전체 실행 시간이 증가**
- Hadoop K-means: **Baseline 대비 3% 증가** → 가장 낮은 변동 (가장 선방)
- Spark PageRank: **약 50% 증가** → 가장 큰 변동

> **이유**: Adaptive는 최소 자원(vCPU 1개, 메모리 2.5GB)에서 시작해 3초마다 조금씩 늘려가는 hill-climbing 방식이므로, 초기 자원이 부족한 동안 병목 발생

### 병렬 컨테이너 실험 (Fair vs Adaptive)

- K-means 벤치마크: **Hadoop에서 56%, Spark에서 12% 메이크스팬 단축**
- 전체 집합 성능: **Hadoop 50.1%, Spark 13.7% 향상**

> **이유**: Fair 방식은 자원을 고정 분배하여 유휴 자원이 낭비되는 반면, Adaptive는 한 컨테이너가 쉬는 동안 다른 컨테이너가 그 자원을 가져다 사용

#### 실험 결과 수치 요약

**단일 컨테이너 실행 결과 (초)**

| 벤치마크 | Hadoop Baseline | Hadoop Adaptive | Spark Baseline | Spark Adaptive |
|----------|-----------------|-----------------|----------------|----------------|
| Terasort | 1275.2 | 1471.2 | 1725.6 | 1848.8 |
| K-means | 1806.2 | 1860.4 | 1151.6 | 1333.0 |
| PageRank | 1285.8 | 1334.0 | 1741.8 | 2607.6 |
| Join | 552.4 | 595.8 | 270.8 | 398.8 |

**병렬 컨테이너 실행 결과 (초)**

| 벤치마크 | Hadoop Fair | Hadoop Adaptive | Spark Fair | Spark Adaptive |
|----------|-------------|-----------------|------------|----------------|
| Terasort | 3992.0 | 5007.6 | 6146.4 | 5140.8 |
| K-means | 10621.2 | 4620.4 | 5605.4 | 4953.2 |
| PageRank | 5432.8 | 5301.0 | 6860.4 | 5918.4 |
| Join | 2002.8 | 2187.8 | 1864.6 | 1868.6 |
| **Total** | **10621.2** | **5301.0** | **6860.4** | **5918.4** |

## Section 4.2 — CPU 친화성(CPU Affinity) 실험

> **실험 구도**: Adaptive (기본) vs **Adaptive-affinity** (CPU 코어 고정 추가 적용)

- **Hadoop**: Adaptive-affinity가 기본 Adaptive 대비 **1.9% 개선**에 그침 → 효과 미미
- **Spark**: Adaptive-affinity가 기본 Adaptive 대비 **5.7% 더 나빠짐**

> **이유**: 빅데이터 워크로드는 compute-intensive하지 않아 코어를 고정하면 오히려 유연성이 떨어짐. 특히 Spark의 K-means, PageRank 같은 반복 연산 작업은 여러 코어를 자유롭게 넘나드는 것이 유리함
